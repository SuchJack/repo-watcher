# 仓库变更监控工具 — 概要设计（方案一：轮询）

**技术栈**：Python（后端）+ Vue（前端）  
**方案**：轮询 GitHub/Gitee 仓库，发现新提交后通过可配置渠道通知用户。

---

## 1. 目标与范围

### 1.1 目标

- 定时轮询配置的 GitHub/Gitee 仓库，检测是否有**新提交**（可选：Release、Issue 等）。
- **轮询频率可配置**：用户可在前端设置轮询间隔（如 5 分钟、10 分钟），保存后立即生效。
- 发现变化后，通过**飞书 Webhook**与**邮箱（用户自行配置 SMTP）**发送通知；所有通知相关配置均在前端填写。
- 提供 Web 界面：配置仓库、查看监控状态、手动触发检查；**全部配置均在网页前端完成，无需手动编辑配置文件**。

### 1.2 范围

- **输入**：仓库列表（平台、owner、repo、分支）+ **可设置的轮询间隔** + 通知方式配置（**飞书 Webhook、邮箱 SMTP**，均通过前端维护）。
- **输出**：有更新时通过飞书/邮箱通知；前端展示各仓库最新 commit、最后检查时间、是否有更新。
- **不包含**：Webhook 实时推送；代码拉取或自动化部署。

### 1.3 配置方式

- **所有配置均通过 Web 前端完成**：仓库列表、轮询间隔、飞书 Webhook、邮箱 SMTP 等均由用户在浏览器中填写并保存，后端通过 API 持久化到文件或数据库。
- 无需直接编辑 `repos.yaml` / `config.yaml`；前端为唯一配置入口。

## 3. 模块划分

### 3.1 后端（Python）

| 模块 | 职责 | 说明 |
|------|------|------|
| **API 层** | 提供 REST 接口供 Vue 调用 | 仓库 CRUD、配置读写、触发检查、查询状态/结果 |
| **调度器** | 定时触发全量检查 | 间隔从配置读取 `poll_interval_seconds`；配置更新时 reschedule 使新间隔立即生效；APScheduler 或 threading |
| **配置加载** | 解析并校验配置 | 仓库列表、轮询间隔、通知配置（飞书/邮箱）、API Token；由 API 读写，持久化到文件或 DB |
| **仓库检查器** | 按平台请求最新 commit | GitHub/Gitee 适配器，统一返回 (sha, message, author, url, time) |
| **状态存储** | 读写「上次 commit SHA」 | JSON 文件或 SQLite，key 为 repo+分支 |
| **通知构造** | 拼装通知文案 | 标题 + 正文（含仓库、分支、commit 信息、链接） |
| **通知分发** | 调用各渠道发送 | **飞书 Webhook**、**邮件（用户自配置 SMTP）**，可插拔；未启用或未配置则不发 |

### 3.2 前端（Vue）

| 模块/页面 | 职责 | 说明 |
|-----------|------|------|
| **仪表盘** | 展示监控概览 | 表格/卡片：仓库、分支、最新 commit、最后检查时间、是否有更新 |
| **仓库管理** | 增删改监控仓库 | 表单：platform、owner、repo、branch；保存时调用 `POST/PUT/DELETE /api/repos`，后端持久化 |
| **通知设置** | 配置通知渠道 | **飞书**：Webhook URL、可选 Secret、启用开关；**邮箱**：SMTP 服务器/端口/TLS、账号/密码、发件人/收件人、启用开关；保存时 `PUT /api/config` |
| **系统配置** | 轮询与开关 | **轮询间隔**（秒或分钟输入 + 校验范围）、是否启用定时检查；保存时 `PUT /api/config`，后端 reschedule |
| **操作** | 手动触发 | 「立即检查」按钮调用后端触发一次全量检查 |

## 4. 数据流

### 4.1 一次轮询（后端）

1. 调度器到点（或 API 触发）→ 执行全量检查。
2. 配置加载：读取仓库列表、通知配置、Token。
3. 对每个 (owner, repo, branch)：
   - 仓库检查器请求 GitHub/Gitee API → 得到当前最新 commit。
   - 状态存储取出该 repo+分支 的上次 SHA。
   - 若当前 SHA ≠ 上次 SHA → 有更新 → 通知构造 → 通知分发发送。
   - 状态存储更新为当前 SHA。
4. 可选：将本次检查结果写入 DB/文件，供前端「最近检查结果」展示。

### 4.2 前端与后端交互

- 打开前端 → 请求 `GET /api/repos`、`GET /api/config`、`GET /api/state`（或合并为一个状态接口）→ 渲染仪表盘与配置表单。
- 用户添加/编辑/删除仓库 → `POST/PUT/DELETE /api/repos` → 后端持久化，**配置仅通过 API 由前端写入**。
- 用户修改**轮询间隔**或**通知设置**（飞书、邮箱）→ `PUT /api/config` → 后端持久化并 reschedule（轮询间隔变更时）。
- 用户点击「立即检查」→ `POST /api/check` → 后端执行一轮检查并返回结果摘要。

## 5. 配置与存储设计

### 5.1 仓库列表（repos）

- 结构示例（YAML/JSON 或数据库表）：
  - `platform`: `github` | `gitee`
  - `owner`, `repo`, `branch`（branch 可选，默认 main/master）
- **仅通过前端 API 增删改**，后端持久化到文件或 SQLite；用户不直接编辑文件。

### 5.2 全局配置（config）

- **轮询**：
  - `poll_interval_seconds`：轮询间隔（秒），如 300、600、900；建议最小值 60（避免 API 限流），可选最大值（如 86400）。
  - `scheduler_enabled`：是否启用定时检查。
  - 配置更新后调度器需 **reschedule**，使新间隔立即生效。
- **API**：GitHub token、Gitee token（可选，提高限流）；可由前端在「系统配置」中填写并保存。
- **通知**（均由用户在前端配置）：
  - **飞书**：`feishu_enabled`、`feishu_webhook_url`（必填时启用）、可选 `feishu_secret`（签名校验）。
  - **邮箱**：`email_enabled`、`smtp_host`、`smtp_port`、`smtp_user`、`smtp_password`、`smtp_use_tls`、`from_addr`、`to_addrs`（列表）；为用户自行配置的 SMTP，不写死任何邮箱。
- **状态存储**：路径或类型（file / sqlite）。
- **读写**：配置由 API 读写，前端为唯一配置入口；敏感字段（如密码、secret）保存时若未修改可保留原值，接口返回时可脱敏（如 `****`）。

### 5.3 状态存储（state）

- **Key**：`{platform}:{owner}/{repo}:{branch}`。
- **Value**：`last_sha`、可选 `last_time`、`last_check_time`。
- **介质**：JSON 文件或 SQLite。

## 7. 外部接口（GitHub / Gitee）

### 7.1 GitHub

- **API**：`GET https://api.github.com/repos/{owner}/{repo}/commits?sha={branch}&per_page=1`  
  Header: `Authorization: token <GITHUB_TOKEN>`
- **RSS**（无需 Token）：`https://github.com/{owner}/{repo}/commits/{branch}.atom`

### 7.2 Gitee

- **API**：`GET https://gitee.com/api/v5/repos/{owner}/{repo}/commits?sha={branch}&per_page=1`  
  参数或 Header 带 `access_token=<GITEE_TOKEN>`

统一抽象为「按 platform 选择实现，返回最新一条 commit 的 (sha, message, author, url, time)」。

## 8. 通知模块

- **消息内容**：标题如 `[仓库更新] owner/repo (branch)`；正文含最新 commit 的 SHA、作者、提交信息、链接。
- **渠道**（本方案实现两类）：
  - **飞书 Webhook**：用户在前端配置群机器人 Webhook URL（及可选 Secret）。请求体按飞书文档格式（如 `msg_type: "text"` 或 `"post"`），将标题与正文填入 `content.text` 或 `content.post`。
  - **邮箱**：用户在前端自行配置 SMTP（服务器、端口、账号、密码、发件人、收件人），后端使用 smtplib/aiosmtplib 发送，不写死任何邮箱地址。
- 每渠道独立「启用」开关；未启用或 URL/收件人为空则不发送该渠道。

## 13. 实现指南

以下按「推荐实现顺序」给出各模块的实现要点，便于按步骤落地。

### 13.1 实现顺序建议

1. **后端：配置与存储** → 能读写的配置/状态，为后续所有逻辑提供数据源。
2. **后端：仓库检查器** → 单仓库「取最新 commit」+ 状态比较，可先不接通知、不接调度，单独测通。
3. **后端：通知模块** → 飞书 + 邮件发送，用固定文案测试。
4. **后端：调度与全量检查** → 串联配置 + 检查器 + 状态 + 通知，支持「手动触发一次」与「定时执行」。
5. **后端：REST API** → 仓库 CRUD、配置 GET/PUT、触发检查、返回状态；配置更新时触发 reschedule。
6. **前端：API 封装 + 基础布局** → 请求后端、路由、布局（如侧栏 + 主内容）。
7. **前端：各页面** → 仪表盘、仓库管理、系统配置、通知设置；表单提交调用对应 API。

### 13.2 后端实现要点

#### 配置与存储

- **配置文件**：使用 JSON 存 `config.json`（轮询间隔、飞书、邮箱、Token）和 `repos.json`（仓库列表），或单文件合并。应用启动时若文件不存在则写入默认配置（空仓库列表、间隔 600、通知未启用）。
- **状态文件**：`state.json`，结构如 `{"github:owner/repo:branch": {"last_sha": "xxx", "last_time": "..."}}`。读/写时加文件锁或单线程写，避免并发覆盖。
- **敏感字段脱敏**：GET /api/config 返回时，将 `smtp_password`、`feishu_secret` 等替换为占位符（如 `"****"`）；PUT 时若前端传占位符则保留原值不覆盖。

#### 仓库检查器

- **抽象**：定义 `CommitFetcher` 接口或协议，方法 `fetch_latest(owner, repo, branch) -> LatestCommit | None`，返回 `(sha, message, author, url, time)`。
- **GitHub**：`httpx.get(f"https://api.github.com/repos/{owner}/{repo}/commits", params={"sha": branch, "per_page": 1}, headers={"Authorization": f"token {token}"})`，解析 `response.json()[0]`。
- **Gitee**：`httpx.get(f"https://gitee.com/api/v5/repos/{owner}/{repo}/commits", params={"sha": branch, "per_page": 1, "access_token": token})`，同样取第一条。
- **状态比较**：key 用 `f"{platform}:{owner}/{repo}:{branch}"`；若 state 中无该 key 或 `state[key]["last_sha"] != new_sha` 则视为有更新，并更新 state。

#### 通知模块

- **飞书**：`httpx.post(webhook_url, json={"msg_type": "text", "content": {"text": 正文}})`；若需签名则按飞书文档用 secret 计算签名并放入 header。
- **邮件**：使用 `smtplib.SMTP`（或 `aiosmtplib` 异步），根据 `smtp_use_tls` 决定 `starttls()`；`from_addr`、`to_addrs` 从配置读取，主题与正文拼成「仓库更新」内容。
- **统一入口**：如 `notify(repo_info, commit_info, config)`，内部根据 `config.feishu_enabled` / `config.email_enabled` 分别调用飞书/邮件；任一方失败记日志，不阻塞另一方。

#### 调度器

- **APScheduler**：使用 `BackgroundScheduler`，添加一个 `interval` 类型的 job，`seconds=config.poll_interval_seconds`，执行「全量检查」函数。
- **全量检查**：读取 repos + config；对每个 repo 调用检查器 → 比较 state → 有更新则通知并写 state；若 `scheduler_enabled` 为 false 可在调度逻辑里直接 return 不执行。
- **Reschedule**：PUT /api/config 时，在更新配置文件后调用 scheduler 的 `remove_job` + `add_job`，或 `reschedule_job`，使新的 `poll_interval_seconds` 生效。

#### API 层

- **GET /api/repos**：读取 repos 列表，再对每条从 state 中取 `last_sha`、`last_check_time`（若 state 按 repo 记录最后检查时间）或全局最后检查时间，一并返回。
- **POST /api/repos**：校验 body（platform、owner、repo、branch），追加到 repos 并写回文件。
- **PUT /api/config**：校验 body（轮询间隔范围、飞书 URL 格式、邮箱必填项等），写回 config；若 `poll_interval_seconds` 或 `scheduler_enabled` 变更则 reschedule。
- **POST /api/check**：直接调用全量检查函数，返回本次检查到的「有更新的仓库」列表或摘要，便于前端展示。

### 13.3 前端实现要点

- **API 封装**：`src/api/repos.js`、`src/api/config.js` 等，使用 axios 请求 `BASE_URL + '/api/repos'` 等；BASE_URL 在开发环境为 Vite proxy 的 `/api` 代理到后端，生产为同源或环境变量。
- **仪表盘**：表格列：平台、仓库、分支、最新 commit SHA/信息、最后检查时间、是否有更新；数据来自 GET /api/repos；顶部有「立即检查」按钮调 POST /api/check，成功后刷新列表。
- **仓库管理**：表单字段 platform（下拉）、owner、repo、branch；提交调用 POST /api/repos；列表每行有编辑/删除，编辑弹窗预填后 PUT /api/repos/:id，删除调用 DELETE。
- **系统配置**：数字输入「轮询间隔（秒）」或带单位（如分钟）转成秒、开关「启用定时检查」；加载时 GET /api/config 填表，保存时 PUT /api/config（body 含 poll_interval_seconds、scheduler_enabled）。
- **通知设置**：分两块——飞书（Webhook URL、Secret 可选、启用开关）、邮箱（SMTP 主机、端口、TLS、账号、密码、发件人、收件人、启用）；保存时 PUT /api/config（body 含 feishu_*、email_* 等）。密码框若后端返回脱敏值，占位显示「留空则不修改」。
- **路由**：如 `/` 仪表盘、`/repos` 仓库管理、`/settings` 系统配置、`/notify` 通知设置；或设置页内 Tab 切换「系统配置」与「通知设置」。

### 13.4 联调与自测

- 后端先单独测：用 curl 或 Postman 调 POST /api/repos 添加一个 GitHub 仓库，再 POST /api/check，看 state 文件是否写入、是否有日志；配置飞书 Webhook 后再次触发检查，看是否收到消息。
- 前端代理到后端（如 Vite `proxy: { '/api': 'http://127.0.0.1:8000' }`），先保证 GET /api/repos、GET /api/config 能拿到数据并渲染，再做增删改与保存。
- 轮询间隔可先设为 60 秒，便于观察定时任务是否按新间隔 reschedule。
