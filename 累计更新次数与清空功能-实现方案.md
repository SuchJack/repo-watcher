# 累计更新次数与清空功能 — 实现方案

本文档描述「最近有更新 = 累计更新次数」以及「每行 / 全局清空按钮」的完整实现方案，便于按步骤开发。

---

## 一、需求摘要

| 需求 | 说明 |
|------|------|
| 1 | **最近有更新** 显示为 **所有仓库的累计更新次数之和**，而不是「有更新的仓库个数」。 |
| 2 | 为每个仓库 **记录累计更新次数**（`update_count`），并支持清空。 |
| 3 | **每行一个按钮**：清空该行对应仓库的更新次数。 |
| 4 | **全局一个按钮**：清空所有仓库的更新次数。 |

---

## 二、后端实现

### 2.1 状态结构扩展

**文件**：`backend/checker/state.py`  
**存储**：`backend/data/state.json`（key 格式：`{platform}:{owner}/{repo}:{branch}`）

**当前 state 单条结构示例**：

```json
{
  "last_sha": "xxx",
  "last_time": "...",
  "last_message": "...",
  "last_author": "...",
  "last_url": "...",
  "last_check_time": "2026-03-08T...",
  "updated": true
}
```

**新增字段**：

- `update_count`（int）：该仓库自上次清空以来的累计更新次数，默认为 0 或不存在时按 0 处理。

**修改**：`check_and_update()` 中，当判定为有更新（SHA 变化）时：

1. 读取 `old = state.get(key, {})`。
2. 计算 `new_count = (old.get("update_count") or 0) + 1`。
3. 写入新 state 时增加 `"update_count": new_count`，并保留 `"updated": True` 及其他现有字段。

无更新时（SHA 未变）：仅刷新 `last_check_time`，不修改 `update_count` 与 `updated`。

---

### 2.2 清空单个仓库的更新次数

**新增接口**：`POST /api/repos/{repo_id}/clear-updates`

**逻辑**：

1. 使用现有 `load_repos()` 根据 `repo_id` 找到仓库（`platform`, `owner`, `repo`, `branch`）。
2. 若未找到，返回 `404`。
3. 使用 `get_state_key(platform, owner, repo, branch)` 得到 state key，`load_state()` 读全量 state。
4. 对该 key 的条目做：
   - `update_count = 0`
   - `updated = False`
   - 保留 `last_sha`、`last_check_time`、`last_message` 等，避免影响后续「是否有新提交」的判断。
5. `save_state(state)` 写回。
6. 返回 `200` 及该仓库最新 state（或 `{"ok": true}`），便于前端刷新。

**实现位置**：`backend/main.py`，在 Repos API 区域新增路由；可复用 `get_repo_state`，再在 config 中增加“根据 repo_id 查找 state key”的辅助（或直接在路由里用 `load_repos` + `get_state_key`）。

---

### 2.3 清空所有仓库的更新次数

**新增接口**：`POST /api/state/clear-all-updates` 或 `POST /api/repos/clear-all-updates`

**逻辑**：

1. `repos = load_repos()`，`state = load_state()`。
2. 对每个 repo 计算 `key = get_state_key(platform, owner, repo, branch)`，若 `key` 在 `state` 中存在，则对该条做：
   - `state[key]["update_count"] = 0`
   - `state[key]["updated"] = False`
   - 其余字段不变。
3. `save_state(state)`。
4. 返回 `200` 及例如 `{"cleared": len(repos)}` 或当前全量 state。

**实现位置**：`backend/main.py`，可在 State API 或 Repos API 区域新增路由；需从 `backend.config` 引入 `get_state_key`、`load_repos`、`load_state`、`save_state`。

## 三、前端实现

### 3.1 统计卡片「最近有更新」

**文件**：`frontend/src/views/Dashboard.vue`

**当前**：

```js
const updatedCount = computed(() =>
  repos.value.filter(r => r.state?.updated).length
)
```

**改为**（所有仓库的 `update_count` 之和）：

```js
const updatedCount = computed(() =>
  repos.value.reduce((sum, r) => sum + (r.state?.update_count ?? 0), 0)
)
```

模板中 `{{ updatedCount }}` 不变，仅数据含义变为「累计更新次数」。

---

### 3.2 表格：新增「更新次数」列（可选）

**文件**：`frontend/src/views/Dashboard.vue`

在「状态」列前增加一列，例如：

- 列标题：**更新次数**
- 单元格：`{{ row.state?.update_count ?? 0 }}`

便于用户看到每个仓库各自贡献的次数。

---

### 3.3 表格：每行「归零」按钮

**文件**：`frontend/src/views/Dashboard.vue`

- 在表格中增加一列「操作」或扩展现有列，放置按钮，文案如 **归零** 或 **清空**。
- 点击后调用 `clearRepoUpdates(row.id)`（见下），成功后执行 `loadRepos()` 刷新列表。
- 可加 `loading` 状态（如按 `row.id` 记录），防止重复点击。

**API**：在 `frontend/src/api/repos.js` 中新增：

```js
export const clearRepoUpdates = (repoId) => http.post(`/repos/${repoId}/clear-updates`)
```

---

### 3.4 操作栏：全局「全部归零」按钮

**文件**：`frontend/src/views/Dashboard.vue`

- 在「立即检查」按钮旁增加一个按钮，文案如 **全部归零** 或 **清空全部更新**。
- 点击后调用 `clearAllUpdates()`（见下），成功后执行 `loadRepos()`。

**API**：在 `frontend/src/api/repos.js` 中新增（若后端路径为 `/api/state/clear-all-updates`）：

```js
export const clearAllUpdates = () => http.post('/state/clear-all-updates')
```

若后端采用 `POST /api/repos/clear-all-updates`，则：

```js
export const clearAllUpdates = () => http.post('/repos/clear-all-updates')
```

前端与后端路径保持一致即可。

---

## 四、接口汇总

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/repos | 不变；响应中每个 repo 的 `state` 需包含 `update_count`（后端 state 写入时已包含）。 |
| POST | /api/repos/{repo_id}/clear-updates | 清空指定仓库的更新次数（`update_count=0`，`updated=false`）。 |
| POST | /api/state/clear-all-updates 或 /api/repos/clear-all-updates | 清空所有仓库的更新次数。 |

---

## 五、数据流与兼容

- **旧 state**：若 state 文件中某 key 无 `update_count`，读写时按 `0` 处理（`old.get("update_count") or 0`），无需迁移脚本。
- **清空语义**：仅重置「未读更新」计数与展示，不清空 `last_sha` 等，下次轮询仍按 SHA 比较，不会误报大量更新。

---

## 六、实现顺序建议

1. **后端**：在 `checker/state.py` 中为有更新分支增加 `update_count` 的累加与持久化。
2. **后端**：在 `main.py` 中实现 `POST /api/repos/{repo_id}/clear-updates` 与 `POST /api/state/clear-all-updates`（或 `/api/repos/clear-all-updates`）。
3. **前端**：修改 `updatedCount` 为累计和，并（可选）增加「更新次数」列。
4. **前端**：在 `api/repos.js` 增加 `clearRepoUpdates`、`clearAllUpdates`，并在 Dashboard 中增加每行按钮与全局按钮及刷新逻辑。

按上述顺序可实现「最近有更新 = 累计更新次数」以及「每行 + 全局」两个清空入口，且与现有轮询、通知逻辑兼容。
